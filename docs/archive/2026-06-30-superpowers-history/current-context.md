# 历史结论摘要

本文从 2026-06-30 之前的 `plans`、`reviews`、`specs` 中提炼仍然有效的信息。后续默认读本文，不再逐个翻旧计划。

## 当前架构事实

- 系统已经转为 ClickHouse-first。数据中心、尾盘实盘筛选、模型审计、行情快照、日线、5 分钟线和基金尾盘数据都应以 ClickHouse 为运行时主数据源。
- Parquet/研究数据集只用于可复现实验，不再作为数据中心的主展示对象。
- Web 后台是控制面：负责展示状态、写配置、触发一次性任务、查看日志，不应该是维持数据更新的唯一进程。
- 长时间运行的数据更新应交给独立 runner，由外部监督机制守护，并写入持久化任务配置、心跳和运行记录。
- 数据中心要优先回答两个问题：数据源现在能不能用；更新任务是否健康到足以持续保持可用。

## 数据质量结论

- 需要一个统一的可交易股票池。数据中心、回测、ML 审计和尾盘实盘不能各自定义股票池。
- ST 判断不能用 `%ST%` 这类粗糙 SQL，应复用项目里的精确定义，避免误判。
- 日线和分钟线质量检查必须覆盖完整度、新鲜度、重复记录、非法 OHLC、成交量和成交额异常。
- 日线修复必须支持跨进程并发安全，不能只靠进程内锁。
- 历史脏 OHLC 和成交额单位漂移会污染研究、回测、ML 标签和图表，必须显式检测。
- 行情快照和 1m/5m 聚合是独立数据源。快照优先模式适合尾盘实时筛选；完整 5m 同步较重，需要明确触发。
- 数据中心页面必须具备局部容错能力。某个可选审计接口失败，不能导致整页空白。

## 数据更新任务方向

当前应重点建设这些任务：

- `post_close_maintenance`：收盘后日线、质量检查和派生数据维护。
- `minute5_intraday_sync`：盘中 5 分钟线同步。
- `quote_snapshot_capture`：行情快照采集。
- `quote_rollup_refresh`：快照到分钟线的聚合刷新。
- `quality_snapshot`：数据质量快照。

每个任务都要有启停配置、心跳、最近开始/结束时间、下一次计划时间、状态、最近结果/错误和影响的数据源。

Web 只负责展示任务状态、修改持久化配置、触发单次运行。真正循环执行任务的是独立 runner。

## 尾盘策略结论

- 尾盘相关系统至少有四条路径：日频因子研究/回测、分钟级实盘筛选、ML 增强排序、基金尾盘流程。
- 四条路径语义不同，不能混成一个指标；但应共享数据质量门禁和股票池定义。
- 简单 IC 容易误导。后续评估因子必须看 walk-forward、序列自相关、扣成本 edge 和 top-N 实际收益。
- `overnight_momentum` 有一定方向性，但可能受自相关和成本侵蚀影响，不能单独当成可交易 edge。
- `tail_session` 的 IC 偏弱，但 top 分位仍可能有价值。实际交易目标是 top-N 选择，不是单纯追求 IC。
- 后续策略优化必须先定义清楚 forward return、交易成本、选股池和基线，避免隐藏过滤 bug。

## 尾盘 ML 结论

- ML 只能作为增强层，不能替代清晰可执行的交易规则。
- 每个模型版本必须记录训练窗口、特征列、标签定义、验证指标、数据质量检查和制品元数据。
- 市场环境、行业环境、风险调整评分、成交额/资金热度都是候选特征，但 smoke 实验只能作为假设来源。
- 审计里要区分“历史长度不足”和“数据质量不合格”。

## 基金尾盘和观察池结论

- 基金尾盘有两个模式：管理已知基金/观察池，以及从候选池发现新机会。
- 观察池应该是可维护数据，不能继续作为静态代码常量。
- 机会发现要先看数据质量、代理指数可靠性、费用、申购赎回约束和品类规则，再做排序。
- 股票观察池是决策辅助，不是自动交易信号；它应说明入场区域、趋势状态和规则原因。

## 数据中心产品结论

- 数据中心不能继续堆成大而全的诊断页。主流程应是：健康矩阵、更新任务状态、高级诊断。
- 按钮必须代表当前可执行动作。数据源健康度 100% 时，不应显示可点击的修复按钮，除非存在明确可执行的修复项。
- 详情面板要说明数据来源、更新机制、系统使用位置、质量规则和修复方式。
- `DataCenter.vue` 和 `app.py` 已经过宽，后续改动应顺手拆分职责。
