# 股票观察池监控设计

## 目标

建设一个后台可见的股票观察池监控模块，用来辅助判断关注股票是否接近合理入场区域。第一版只做本地后台分析和状态展示，不做主动通知、不下单、不自动修改券商或第三方观察池。

## 范围

观察池覆盖一组可配置股票，系统每天为每只股票生成结构化分析。初始股票包括：

| 代码 | 名称 | 主题 |
|---|---|---|
| `000636` | 风华高科 | MLCC / 被动元件 |
| `002859` | 洁美科技 | MLCC 耗材 / 载带 |
| `300285` | 国瓷材料 | MLCC 上游材料 |
| `300408` | 三环集团 | MLCC / 陶瓷元件 |
| `300014` | 亿纬锂能 | 电池 / 储能 |
| `688017` | 绿的谐波 | 机器人 / 减速器 |
| `601899` | 紫金矿业 | 金铜资源 |

第一版支持手动刷新和后端 API 刷新。定时任务和推送通知暂不做，但分析模型应暴露可复用的触发状态，方便后续接通知渠道。

## 用户流程

1. 用户打开后台，进入观察池页面。
2. 页面展示关注股票表格。
3. 每行展示最新价、日涨跌、5 日收益、20 日收益、量能状态、监控状态、距离入场区域的距离。
4. 用户展开详情，查看简短分析、关键价位和触发规则。
5. 用户把状态作为决策辅助，而不是自动交易建议。

## 配置

第一版使用 `config/watchlist_monitor.yaml` 作为配置源。每只股票包含：

- `symbol`：A 股代码。
- `name`：展示名称。
- `theme`：业务或市场主题。
- `notes`：关注逻辑。
- `levels`：观察、入场、加仓、失效、突破确认等人工价位。

人工价位是刻意设计：系统不假装能从历史价格自动推导完美买点，而是把用户定义价位与客观趋势、量能信息结合。

## 后端设计

新增 `src/monitoring/watchlist.py`，提供纯分析函数和数据模型：

- `WatchlistConfig`
- `WatchlistStockConfig`
- `WatchlistLevels`
- `WatchlistStockSnapshot`
- `WatchlistReport`

核心职责：

- 规范化股票代码。
- 通过现有数据源读取最新行情和日线。
- 计算 5 日、20 日收益。
- 计算 5 日、20 日均线。
- 通过最新成交量和近期均量比较判断量能变化。
- 结合人工价位和趋势状态生成监控状态。

监控状态：

| 状态 | 含义 |
|---|---|
| `near_entry` | 接近入场区 |
| `observe` | 继续观察 |
| `extended` | 短期偏高，避免追高 |
| `weak` | 趋势偏弱 |
| `invalidated` | 跌破失效位 |
| `missing_data` | 数据不足 |

## 前端设计

新增观察池页面，沿用后台现有 Element Plus 风格。页面包含：

- 状态概览。
- 股票监控表。
- 详情展开区。
- 手动刷新按钮。
- 数据不足或刷新失败提示。

## 分析规则

- 接近入场区：价格落入或接近 `entry` 区间，且未跌破 `invalid`。
- 偏高：价格明显高于 `breakout` 或距离入场区过远。
- 趋势偏弱：价格低于关键均线且量能没有改善。
- 失效：价格低于 `invalid`。
- 数据不足：缺少最新价或日线样本不足。

## 错误处理

- 单只股票数据失败时，只标记该股票 `missing_data`，不影响整页。
- 配置文件缺失时返回空观察池和清晰错误信息。
- 行情接口失败时保留页面结构，显示失败原因。

## 测试计划

- 测试配置解析。
- 测试每一种监控状态分类。
- 测试缺失数据不会导致整体失败。
- 测试 API 返回观察池报告。
- 测试页面包含关键状态、价位和详情原因。

## 非目标

- 不接券商交易。
- 不做自动下单。
- 不做主动通知。
- 不从历史价格自动生成买点。
- 不替代用户判断。

## 验收标准

- 后台可以看到观察池表格。
- 每只股票有状态、关键价位、趋势和量能说明。
- 数据缺失时页面不崩溃。
- 观察池配置可通过 YAML 维护。
