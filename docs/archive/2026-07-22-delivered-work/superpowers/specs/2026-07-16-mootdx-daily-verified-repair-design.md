# Mootdx 日线核验后精准回补设计

## 目标

在 Baostock 对连续日线缺口块完成核验后，只回补被确认存在日线的交易日，自动排除确认无数据的日期，避免用户在当前选中日期逐日误操作。

## 数据契约

日线质量接口的缺口明细需要在完整块范围内暴露每个交易日的最新核验 verdict：`available`、`no_data`、`error` 或未核验。保留已有 `missing_dates`（完整连续块）和 `block_missing_dates`（前端展示模型），新增可用于动作决策的日期分组，或前端从接口逐日 verdict 构建分组；不得将周末/节假日加入范围。

## 页面交互

当某个连续缺口块至少有一个 `available` verdict 时，显示“创建精准回补（N 日）”。点击后只提交这些 `available` 日期；可一次提交多个单日回补项。已确认 `no_data` 的日期计入“无需回补”，不出现在回补请求中。存在 `error` 或尚未核验的日期时，在页面保留“待核验”提示，不能被当作可回补。

以 `002005.SZ` 为例，06-03 至 07-13 的块中 27 日为 `available`、07-13 为 `no_data`：页面显示“27 日可回补 / 1 日无需回补”，一次回补 27 日，并排除 07-13。

## 后端行为

复用现有定向回补 endpoint 和单日 `start_date/end_date` 语义；客户端为每个 `available` 日期创建一个回补 item。后端保持对每个 item 的日期范围校验，不在服务端推测 Baostock verdict，避免修改既有回补任务职责。

## 验收

- 质量 API 对一个缺口块明确暴露逐日核验状态，且只包含该块缺失交易日。
- 前端精准回补 payload 只含 `available` 日期，绝不含 `no_data`、`error` 或未核验日期。
- 27 个可用日期产生 27 个单日回补 item；07-13 `no_data` 不产生 item。
- 无可用日期时，精准回补按钮禁用或不显示，并保留核验提示。
- 既有手动单日回补、块级 Baostock 核验、旧数据源体系不受影响。
