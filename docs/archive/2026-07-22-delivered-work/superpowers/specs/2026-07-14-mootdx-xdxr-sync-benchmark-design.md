# Mootdx XDXR 同步基准与落库设计

## 目标

为 Mootdx 独立数据源建立可重复的除权除息（XDXR）同步基准，并在确认整链路性能和数据语义后，再创建专属同步任务。

本设计只覆盖 `mootdx_*` 表和 Mootdx 客户端；不读取、不写入、不改动 `tdxrs`、`xdxr_info` 或既有复权服务。

## 已验证的网络基准

2026-07-14 对当前 Mootdx 股票目录的 300 只分层样本进行了只读测试。样本覆盖沪市主板 102 只、深市主板 102 只和创业板 96 只；每组复用一个客户端，`timeout=10`，不写入 ClickHouse。

| `rate_limit` | 非空成功 | 异常 | 吞吐 | P95 | 总耗时 |
| --- | ---: | ---: | ---: | ---: | ---: |
| `0` | 300 / 300 | 0 | 20.56 请求/秒 | 54.51ms | 14.59s |
| `0.02` | 300 / 300 | 0 | 22.48 请求/秒 | 47.87ms | 13.34s |

300 只共返回 13,909 条事件，平均 46.36 条/股。以当前目录的 5,203 只股票作线性估算，网络拉取约需 3.9 分钟；该估算不包含解析、去重、批量写入和失败重试。

`bestip=True` 在当前环境的启动测速超过 12 秒仍未完成，因此日常任务必须关闭；不设置固定服务器，直到有可重复的服务器对照证据。

## 基准配置

后续整链路基准和任务候选配置为：

```json
{
  "rate_limit": 0.02,
  "timeout": 10,
  "bestip": false
}
```

`timeout=10` 已覆盖本次 300 只成功样本，但尚未覆盖网络故障情形。若上线前的故障注入或真实长尾测试显示误超时，改回 `15`，不改变 `rate_limit` 和 `bestip` 决策。

## 数据模型

`mootdx_xdxr` 继续作为独立事实表，保存源端事件，不直接参与复权计算。事件主键语义为：

```text
(symbol, event_date, category)
```

同日不同 `category` 是不同事件，必须并存；源端整段历史返回时应以该键覆盖同一版本事件，并保留 `ingested_at` 和 `raw_json` 作为采集与审计证据。

数值字段必须保持空值语义：`suogu`、股本变动字段等源端空值写入 `Nullable(Float64)`，不能转换为 `0`。`raw_json` 保留所有未建模字段（包括 `fenshu`、`xingquanjia`）。

## 同步流程

1. 从 Mootdx 股票目录取得当次有效股票池，并复用单一 `MootdxSource` 客户端。
2. 逐标的调用 `fetch_xdxr(symbol)`；无异常的空 DataFrame 记录为 `empty`，不等同于请求失败。
3. 校验年、月、日可组成有效 `event_date`；非法事件计数并跳过。
4. 将字段归一为可空值，按 `event_date` 月份聚合批量写入 `mootdx_xdxr`。
5. 为每个标的记录状态、事件数、请求耗时和异常类别；为整次运行记录建连、请求、解析、写库和总耗时。
6. 连续三次传输异常时中止本次运行，保留已经写入的审计结果，交给下一次任务重试未完成标的。

## 整链路验收

首次落库基准使用相同的 300 只固定分层样本，随后才允许全量 bootstrap。

- 300 只请求成功率（成功或明确空结果）为 99% 以上，且没有连续三次传输异常。
- 单次运行分别输出建连、请求、解析、写库、总耗时，不能只报告总时长。
- 所有含空值的源字段在查询结果中仍为 `NULL`；测试须覆盖 `suogu` 为空的事件。
- 同日多类别事件均保留；重复运行后 `FINAL` 查询的事件键不重复。
- 全量 bootstrap 的网络拉取预期在 5 分钟量级；写库后的实际耗时、事件数、空结果数、失败数必须被审计，而非用本次估算替代。

## 非目标

- 不修改 `xdxr_info`、`tdxrs` 或 `AdjustmentService`。
- 不根据 `fenhong`、`songzhuangu`、`peigu` 等字段计算复权因子；字段单位和价格联动尚未验证。
- 不在本阶段启用并发请求、固定服务器或 `bestip`。

## 目录驱动的日线同步保鲜策略

日线同步使用目录状态，而不是永久保留曾出现过的代码。目录同步本身每天执行，成本远低于逐标的日线请求，因此承担重新出现检测；休眠代码不需要日线轮询。

| 状态 | 进入条件 | 日线同步 | 退出条件 |
| --- | --- | --- | --- |
| `active` | 当前成功目录快照包含该标的 | 是 | 成功目录快照首次缺失时进入 `pending_removal` |
| `pending_removal` | 连续缺失计数为 1 | 是 | 再次出现则回到 `active`；第二次成功目录快照仍缺失则进入 `dormant` |
| `dormant` | 连续 2 次成功目录快照缺失 | 否 | 任一成功目录快照再次包含该标的时立即回到 `active` |

目录总量相对上次成功快照下降超过 2% 时，该轮视为异常快照：记录审计和告警，但不增加任一标的的缺失计数，也不触发状态迁移。恢复后的目录快照才重新参与连续缺失判断。

目录事实表保存 `is_active`、`missing_catalog_runs`、`last_seen_at`、`deactivated_at` 和 `reactivated_at`。休眠前写入状态行而不是删除历史；重新出现时写入新状态行，因此 `FINAL` 始终表达当前目录状态，并保持完整审计链。
