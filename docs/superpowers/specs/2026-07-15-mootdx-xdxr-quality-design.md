# Mootdx XDXR 质量页设计

## 目标

为 Mootdx 独立 XDXR 同步提供一个日常运维页面，快速判断任务健康度、定位单次运行和逐标的异常，同时保留最小的数据覆盖摘要。

页面只读取 `mootdx_*` 表；不读取、不写入、不展示 `tdxrs`、`xdxr_info` 或既有复权服务。

## 入口与范围

- 新增独立路由：`/data/mootdx/xdxr-quality`。
- 在 Mootdx 监控页新增“XDXR 质量”跳转入口，与“目录质量”“日线质量”并列。
- 默认展示最近 30 次 XDXR 运行；支持按日期范围和运行状态筛选。
- 本阶段不提供重跑、修复或手工写入操作，页面保持只读。

## 页面结构

### 首屏运维概览

展示最近一次 `xdxr` 运行的：

- 运行状态、开始和结束时间；
- 请求标的数、成功数、空结果数、错误数、事件行数；
- 是否触发连续错误熔断；
- 请求、解析、写入和总耗时；
- 失败标的样本。

无运行记录时显示“尚无 XDXR 运行记录”，不视为失败。

### 近 30 次趋势与运行表

趋势按运行时间呈现状态和总耗时。运行表包含时间、状态、成功/空/错误、事件数、总耗时和熔断标记；状态筛选值为全部、成功、失败。

选择一行打开运行明细抽屉。

### 运行明细

抽屉显示该 `run_id` 的 `mootdx_xdxr_symbol_runs`：标的、状态、事件数、请求/解析耗时、错误文本、源字段名。支持按 `success`、`empty`、`error` 筛选，错误信息完整展示但不提供写入动作。

### 数据辅助摘要

展示 XDXR 事实表的覆盖标的数、事件总数、最近采集时间和 `suogu is null` 数量。该摘要仅协助判断数据规模，不能替代运行审计。

## 后端接口与数据流

`MootdxQualityService` 增加独立 XDXR 查询方法，复用现有 ClickHouse 客户端与安全查询约定。

- `GET /api/data/mootdx/xdxr-quality`：返回最新运行、最近运行列表、数据摘要；接受 `limit`（默认 30，上限 100）、`start_date`、`end_date`、`status`。
- `GET /api/data/mootdx/xdxr-quality/runs/{run_id}`：返回该次运行汇总和逐标的审计；接受可选 `status` 和 `limit`（默认 500，上限 1000）。

运行汇总以 `mootdx_sync_runs.task_key = 'xdxr'` 为准，诊断字段从其 `result_json` 读取；逐标的明细以 `mootdx_xdxr_symbol_runs final` 的 `run_id` 关联。运行记录缺少诊断字段时，API 返回安全默认值而不是抛错。

## 错误处理与验收

- ClickHouse 不可用时沿用现有后端错误响应，不返回伪造健康状态。
- 无历史运行、无匹配筛选、某次运行没有审计行，均返回结构稳定的空集合与明确空状态。
- API 测试覆盖聚合、日期/状态筛选、空状态和运行明细。
- 前端测试覆盖路由、监控入口、概览状态、运行表和审计抽屉关键文案。

## 非目标

- 不创建新的同步或修复任务。
- 不重新计算复权因子，也不对事件字段单位下结论。
- 不改动 `tdxrs -> xdxr_info` 及相关页面。
