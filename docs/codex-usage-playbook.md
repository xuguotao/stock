# Codex 使用规范

本文定义本仓库使用 Codex 的协作方式。目标是让每项工作有明确边界、可追溯代码状态和可验证交付，避免多对话、多分支、多目录造成混乱。

本文讨论的是 **如何使用 Codex**，不替代项目架构、数据字典或业务实施方案。

## 1. 基本原则

1. **对话用于协作，Git 用于保存正式代码状态。** 对话内容不是长期事实来源；已确认的设计、代码、测试与运行方式必须落到仓库、提交记录或系统审计中。
2. **一个对话只交付一个可验证目标。** 目标必须能用一句话说明，并能给出明确验收方法。
3. **同一写入面只有一个负责人。** 两个实施对话不能同时修改同一组文件、同一张数据库表、同一个部署配置或同一条数据任务。
4. **`main` 是唯一正式代码状态。** 功能完成后验证、合入 `main`、清理临时分支；不要长期堆积“也许以后再处理”的分支和 worktree。
5. **进度与运行状态分层。** Codex 对话负责说明开发进度；长期数据任务、发布与同步的真实状态必须由后台和数据库审计承载。

## 2. 对话的三种角色

### 2.1 总控对话

每个项目保留一个总控对话，负责：

- 确定当前优先级和任务顺序；
- 将大型目标拆成独立专题；
- 审核实施结果、测试证据和发布影响；
- 决定合入 `main`、部署、重启或数据迁移；
- 保持对“当前正式状态”的简明认知。

总控对话不应长期承担大量无关代码修改。它是决策与收口入口。

### 2.2 专题实施对话

每个实施对话只负责一个独立交付物，例如“增加某个页面”“修复某个同步失败”“为一个数据层增加验收”。

实施对话负责：

- 阅读相关代码和文档；
- 说明影响范围与实施计划；
- 实现、测试、记录必要文档；
- 提供可审查提交和验收方法。

完成后应回到总控对话收口；不要把已经完成的专题对话直接改作下一项无关工作。

### 2.3 只读审查/诊断对话

用于代码审查、数据核验、故障定位、方案比较和风险分析。默认不修改代码、不执行写数据库操作。

这类对话可以与实施对话并行，只要不改变同一写入面。若审查结论需要修复，应新建或切换到明确的实施任务。

## 3. 启动一个实施任务

每个任务开始时，应使用以下最小任务说明：

```text
目标：
完成什么可验证结果。

范围：
允许修改哪些模块、表或页面；明确不修改什么。

验收：
测试命令、数据库验证 SQL、页面操作或输出文件。

权限边界：
是否允许建表、迁移、全量同步、部署、重启、推送。
```

如果任务横跨多个独立目标，应先拆分。一个任务可包含代码、测试和对应文档，但不应同时混入不相关的功能重构。

## 4. 分支与 worktree 规范

### 4.1 默认策略：普通分支

默认工作流：

```text
main
  → 创建 feature/<简短任务名> 分支
  → 一个实施对话完成任务
  → 测试与审查
  → 合入 main
  → 删除 feature 分支
```

普通分支适用于绝大多数任务。它成本最低、最容易理解。

### 4.2 何时使用 worktree

worktree 不是“更高级的分支”，而是让多个分支能同时在不同目录中检出的工具。只有同时满足以下条件才使用：

- 确实有两个或以上实施任务需要并行推进；
- 它们不会修改相同文件、数据库迁移或部署配置；
- 需要在不切换主目录分支的情况下同时运行、调试或测试。

否则不要创建 worktree。一个任务完成并合入后，应立刻删除对应 worktree 和分支。

### 4.3 禁止事项

- 不让两个实施对话写同一个分支；
- 不在未核对工作区是否干净时切换分支、合并或清理；
- 不把“存在 worktree”误认为“代码已经合入”；
- 不在没有验证内容等价或已合入时强制删除分支；
- 不长期保留已经合入的临时 worktree。

## 5. Codex 的工作节奏

### 5.1 实施前

Codex 应先说明：

- 将要修改的范围；
- 是否触及数据库、任务调度、外部服务或部署；
- 如何验证结果；
- 需要用户明确授权的高影响操作。

对于纯只读分析，不需要创建分支或 worktree。

### 5.2 实施中

Codex 应只在状态发生变化时更新进度，例如：

- 已确认原因或设计；
- 已开始执行耗时任务；
- 测试失败；
- 数据同步结束；
- 发现需要用户决策的边界。

不要用重复的“仍在进行中”刷屏。对话进度不能替代真实任务监控。

### 5.3 实施后

每次交付应给出：

- 结果摘要；
- 修改的文件或数据库对象；
- 已运行的验证及结果；
- 需要用户执行的后续动作（如部署或重启）；
- Git 提交、合入状态和工作区状态。

## 6. 数据库、同步和发布类任务

此类任务的风险高于普通代码修改，应额外遵守：

1. 先说明输入、输出、影响表和读取口径；
2. 先做只读检查，再执行写入；
3. 不删除表、历史数据或发布版本，除非用户明确授权；
4. 新版本发布必须保留旧版本可读或可回滚的路径；
5. 长时间同步必须写入可查询的运行审计；
6. 不能仅凭 Codex 对话声明成功，必须给出数据库核验方法；
7. 手工命令与定时任务应尽可能复用同一运行审计和状态模型。

对于“后台是否能看到进度”的需求，应优先建设真实后台状态与审计，而不是依赖对话持续更新。

## 7. Git 收口清单

实施任务进入收口前，依次检查：

```text
[ ] 工作区无无关未提交修改
[ ] 已运行与改动范围相符的测试
[ ] 已检查测试输出和退出码
[ ] 已检查 git diff --check
[ ] 已说明数据库/部署影响
[ ] 已提交清晰、可回退的 Git commit
[ ] 已合入 main 或明确说明未合入原因
[ ] 已清理已合入的临时分支与 worktree
```

如果使用 cherry-pick 合入，应额外比较分支与 `main` 的最终文件内容。提交 ID 不同不等于功能未合入。

## 8. 长期指导文件的分层

| 载体 | 适用内容 |
|---|---|
| 当前对话 | 一次性目标、临时限制、即时决策 |
| `AGENTS.md` | 每次进入仓库都要遵守的简短硬规则 |
| `docs/` | 设计、运行手册、数据字典、验收说明 |
| 项目 skill | 重复且步骤较多的流程，如发布检查、数据变更流程 |
| MCP | 数据库、代码托管、文档、监控等外部系统能力 |
| automation | 定期检查、提醒、长任务状态监控 |

`AGENTS.md` 应保持短小且可执行；不要把完整设计文档复制进去。官方建议将重复出现的项目规则沉淀为 `AGENTS.md`，将可复用复杂流程封装为 skill，并通过 MCP 连接外部系统。[Codex 自定义文档](https://developers.openai.com/codex/concepts/customization)

## 9. 推荐的最小协作模型

对于当前项目，默认只维持：

```text
1 个总控对话
1 个正在实施的专题对话
0～1 个只读审查/诊断对话
```

如无真实并行需求，不增加更多实施对话、分支或 worktree。这样能获得足够的任务隔离，同时保持项目状态容易理解。

## 10. 实施对话的交付模板

```text
结果：
一句话说明是否完成。

改动：
- 文件/模块：……
- 数据库对象：……（没有则写“无”）

验证：
- 命令或操作：……
- 结果：……

Git：
- 分支：……
- 提交：……
- 是否已合入 main：……

后续：
- 需要部署、重启、人工验证或无后续动作。
```
